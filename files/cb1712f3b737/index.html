AIセキュリティポリシー判定ツール

税理士事務所向け ─ Claude Code / 生成AIに「どの情報をどこまで預けるか」をYES/NOで決め、ストレージ構成と設定まで導きます。

はじめに ─ このツールの考え方

「方針メモ」で終わらせず、決めたら設定に落とせる判定表です。YES/NOを選ぶと、末尾の「結果」タブにポリシー一覧・ストレージ推奨・Claude Codeの設定例（settings.json）が自動で組み上がります。

最重要の前提：テレメトリ（利用統計）は停止できますが、AIへの入力（会話・読ませたファイル本文）は処理のためAnthropicのAPIに送信されます。これは仕様です。だからこそ「何を読ませるか／何のフィールドを上げるか」のポリシーが要になります。

※ モデル学習への利用は、Team / Enterprise プランでは原則使用されません（最終的な可否はadmin設定・契約条件で要確認）。テレメトリ停止は別物で、本ツールで設定します。

進め方（4ステップ）

STEP1 ポリシー判定：事故シナリオを見ながら、12+のポリシーをYES/NOで決める

STEP2 フィールド粒度：請求書・通帳などを「項目（フィールド）」に分解し、上げる／マスクして上げる／上げない（隔離）を選ぶ

STEP3 ストレージ判定：クラウド（Google Drive / OneDrive / Dropbox 等）の選択、ローカル＋gitとGitHubの使い方を決める

STEP4 結果：採用ポリシー・運用ルール・推奨構成・settings.jsonを出力（印刷・コピー可）

STEP1 ポリシー判定

各問いに「事故シナリオ」を添えています。YES＝そのポリシーを採用（制限する／設定する）。迷ったら畠山推奨のままでOK。

STEP2 フィールド粒度の判定

書類を「項目（フィールド）」に分解し、Anthropicのサーバーに上げる／マスクして上げる／上げない（隔離）を選びます。例：請求書の「口座情報」は上げない、「取引先名」はマスク、など。

3つの処分： 上げる そのまま読ませてよい ／ マスク 加工・仮名化した版のみ読ませる ／ 上げない 隔離フォルダに退避しClaudeから物理的に除外

STEP3 ストレージ判定

STEP1〜2で決めた「上げない情報」は、そもそもどこに保管するかまで決めないと守れません。クラウド可否とローカル／GitHubの使い方を選びます。

前提：Claude Code は「ローカルのファイル」を前提に動きます。クラウド直下（Drive / OneDrive 同期フォルダ）での起動はフリーズ・git破損の報告があり非推奨。作業実体はローカル、クラウドは保管・共有レイヤーが基本です。

解説 ─ なぜ「ローカル作業＋GitHub」なのか（Drive・Boxだけでは足りない理由）

非エンジニアの感覚だと「データはGoogle DriveやBoxに入れておけば、共有もバックアップもできる。これで十分では？」と思いがちです。でもAIに仕事をさせる前提だと、Drive・Boxだけでは届かない部分があります。鍵は「3つのレイヤーは役割が違う」こと。

レイヤー役割なぜ必要か
ローカル（PC内）AIの作業場Claude Code はローカルファイルを直接読み書きして動く設計。速くて安定。
Drive / Boxデータの保管・共有顧客資料・成果物を全スタッフで共有し、バックアップする。
GitHub仕組みの共有・履歴CLAUDE.md・skills・スクリプト＝事務所のルールを全PC・全員に配り、改善を同期する。

■ なぜローカルで作業するのか

Claude Code は「ローカルのファイル」を直接読み書きしながら動きます。Drive・Boxの同期フォルダ直下で動かすと、AIの高速な読み書きに同期が追いつかず、フリーズ・mkdirエラー・git破損が起きます（実報告多数）。だから作業実体はローカルに置き、Drive・Boxは「保管・共有」に徹するのが安定運用です。

■ なぜDrive・Boxだけでは足りないのか（AI運用の観点）

Drive・Boxの限界中身
版管理が世代バックアップ止まりAIが1回の作業で「何を・なぜ変えたか」を論理的なまとまりで追えない。「この変更だけ取り消す」ができない。
大量変更に弱いAIは一度に大量のファイルを書き換える。Drive・Boxはこれを「同期の嵐・コンフリクトコピー乱立」として扱い、どれが正本か分からなくなる。
検証・巻き戻しができないAI運用の肝は「AIがやったことを後から検証し、まずければ戻す」。これにはgitの履歴（誰が・いつ・何を・なぜ）が要る。Driveの版履歴では代替できない。

■ なぜ、さらにGitHubなのか

GitHubで得られること中身
仕組みを全PC・全員で同期CLAUDE.md・skills・スクリプトを、自宅PC・事務所PC・スタッフ全員で常に最新に揃えられる。
属人化を解く「誰がいつルールを変えたか」が残る。改善を一人が直せば全員に配れる。
データと仕組みを分けて守る顧客データはDrive、仕組みはGitHub。置き場を分けることで、データを公開リスクにさらさず「仕組みの共有」だけを得られる。

■ 「仕組みなら、Driveの共有フォルダに置けば配れるのでは？」への答え

CLAUDE.md・skills・スクリプトも結局はただのファイル。だったらDriveの共有フォルダに入れて全員に配ればいい、と思うのが自然です。でも「全PCで同じく動く仕組みを配り、改善し続ける」目的だと、Driveの同期では次の点で届きません。

Driveで仕組みを配る限界中身
置き場所が合わない仕組みファイルは各PCの決まった場所（~/.claude/ や プロジェクトの .claude/）に無いとClaudeが読まない。Driveは「Driveフォルダ」に同期するだけで、実際に使う場所には届かない。結局、手でコピーする手間が残る。gitは clone / pull で使う場所に直接配れる。
同時編集で壊れる2人が同じCLAUDE.mdを直すと、Driveは後勝ち上書きか「コンフリクトコピー」を量産し、どれが正本か分からなくなる。gitは2人の変更を行単位で統合できる。
確認せず全員に伝染するDriveは保存した瞬間に全員へ即同期。誰かのうっかりや実験中のルールが、レビューなしで全PCへ一気に広がる。gitなら「プルリクで一度確認してから反映」が仕組みで挟める。
一部だけ戻せないスクリプト更新で不具合が出ても、Driveは世代を丸ごと戻すしかなく、巻き添えで他の改善まで消える。gitは「その変更だけ」取り消せる。
半端な状態が混ざる仕組みは複数ファイルが揃って初めて正しく動く。Driveは1ファイルずつ同期するので、PCによって「新CLAUDE.md＋旧skill」のちぐはぐが起きる。gitは一式をまとめて反映するので、全PCが常に整合する。
要は、「データの共有」はDriveで十分でも、「全PCで同じく動く仕組みの配布・改善」はDriveの同期では担えない。同じ「共有」でも、データの保管と仕組みの配布は求められる仕事が違う。だからこそ、仕組みだけはGitHubに置く意味がある。
まとめ：ローカル＝速くて安全な作業場、Drive・Box＝データの保管・共有、GitHub＝仕組みの共有と履歴。役割が違うから、「Drive・Boxに入れるだけ」では作業の安定性とAIの検証可能性は手に入りません。「データはいつものフォルダ（Drive・Box）、AIの作業はローカル、仕組みだけGitHub」と覚えれば、非エンジニアでも線引きを間違えにくくなります。

解説 ─ ローカルの成果物をDriveの「正本」にする手順

「ローカルで作業、Driveで共有」と決めても、完成したファイルをどうDriveに上げれば正本（オリジナル）が崩れないかでつまずきがちです。大原則はひとつ ─ 正本はDrive、ローカルは作業中の写し。逆にすると二重管理でどれが最新か分からなくなります。

■ フォルダ構造のイメージ（3つのレイヤーは同じ並びで揃える）

ローカルもDriveも「顧問先 ＞ 会社 ＞ 年度」の形を揃えるのがコツ。違うのは中身だけ ─ ローカルは作業中・全部入り、Driveは完成版（正本）だけ。

■ ローカル → Drive へ上げる3つのやり方（事務所規模で選ぶ）

やり方中身向き
A. 手動アップ
（最小・一番安全）		完成版だけをDriveのブラウザ画面にドラッグして上げる。作業フォルダとDrive同期フォルダは別に置く。0〜数人。設定不要
B. 同期フォルダ分離Drive Desktopは入れるが、Claudeの作業フォルダは同期対象の外に置く。完成したら作業フォルダ→同期フォルダにコピーして反映（Claudeは同期フォルダの中で動かさない）。共有が多い中規模
C. Claude／APIで直接アップ
（おすすめ）		同期に頼らず、Claude Codeから完成ファイルをDrive APIで直接上げる。同期フォルダ問題（フリーズ・コンフリクト）を根本回避。「完成→指定フォルダへアップ→記録」まで定型化できる。自動化したい人

■ 正本を崩さない4つのルール（どのやり方でも共通）

ルールなぜ
命名を固定する例 申告済み_申告書一式_R7_A社.pdf。別名コピーの乱立を防ぐ。
上書きは「同名で上げる」Drive側がバージョン履歴で旧版を持ってくれる。_最新2 のような別名を作らない。
完成したらローカル作業版は消すか _作業中 を付ける「Driveが正本」と一目で分かる状態に。どっちが本物か迷わない。
ローカルとDriveでフォルダの形を揃える同じ「顧問先＞会社＞年度」。迷子・取り違えを防ぐ。
覚え方：「Driveに上がった版が正本、ローカルは作業中の写し」。完成したものだけをDriveに上げ、同名で上書きする。これだけで「どれが最新か分からない」事故はほぼ消えます。

フォルダ構造の設計

事務所のフォルダの並べ方を決めます。ここでの選択に合わせて、STEP4「結果」の『あなたの構成』フォルダ図が自動で組み上がります。畠山事務所の実運用（番号プレフィックス・分類記号・決算期フォルダ）を初期値にしています。

マークダウン（ルール文書）の保管・更新ルール

CLAUDE.md などのルール文書（マークダウン）を「誰が・どこで・どう更新するか」を決めます。ここを決めて初めて、全PC・全スタッフが同じルールで動くAIになります。選択はSTEP4「結果」⑥に保管・更新マップとして反映されます。

解説 ─ GitHubに「顧客データごと」載せる運用について

先に結論。GitHubは使います。ただし載せるのは仕組み（CLAUDE.md・skills・スクリプト）だけ。顧客の実データ（決算・申告・個人情報）は載せません。
このセクションで「勧めない」と言っているのは、世の中で見かける「顧客データごとGitHubに入れる運用」のこと。GitHubそのものを否定しているわけではありません。

なぜ「顧客データごと」の運用が存在するのか（メリット）、それでも勧めない理由、そして非エンジニアの税理士が使う上でのハードルを整理します。

■ なぜ「顧客データごと」載せる運用が存在するのか（メリット）

メリット中身
バージョン管理誰がいつ何を変えたかを全て記録し、任意の時点に巻き戻せる。
複数人での共有同じデータを全スタッフが常に最新で参照し、変更を取り込み合える。
バックアップ履歴ごとクラウドに保全。PCが壊れても失わない。
変更レビュープルリクエストで「公開・反映の前に一度チェック」が仕組みで回る。
監査証跡変更履歴がそのまま「誰が触ったか」の証跡になる。
ただし、これらのメリットは「仕組み（CLAUDE.md・skills・スクリプト）」だけをGitHubに置けば得られます。顧客の実データ（決算・申告・個人情報）まで載せる必要はありません。データの共有はGoogle Drive等、バックアップはDrive/NASで代替できます。つまり、メリットを取るのにデータごと載せる理由がない。

■ それでも「顧客データごと」は勧めない理由

リスク中身
公開事故publicリポジトリの誤設定・.gitignore漏れで、顧客の決算書やマイナンバーが全世界に公開された事例は実在。一度公開＝検索・キャッシュに残り、取り消せない。
第三者への預託GitHub（Microsoft）のサーバーに顧客データを預けることになる。守秘義務・個人情報の第三者提供の整理が別途必要。
消去が困難gitは「履歴を残す」のが本質。一度commitした機密は、履歴を書き換えない限り過去版に残り続ける。

■ 非エンジニアの税理士にとってのハードル

ハードル中身
概念の壁commit / push / pull / branch / conflict / .gitignore など、日常業務にない概念を覚える必要がある。
誤操作リスク「何を載せて何を載せないか」の線引き（.gitignore）を一度でも間違えると漏洩に直結。専門外の人ほど起きやすい。
継続コストコンフリクト解消・履歴管理でつまずいた時、事務所内に聞ける人がいないと業務が止まる。
結論：GitHubは使う。ただし「仕組み（CLAUDE.md・skills・スクリプト）の共有」だけ。顧客の実データは載せない。データの共有はDrive、バックアップはDrive/NAS。これなら非エンジニアでも「データはいつものフォルダ、仕組みだけGitHub」と役割が明確で、事故りにくい。上のハードル（概念・誤操作・継続コスト）も、git操作自体をClaude Codeに任せればコマンドを覚える必要はほぼなく、越えられます。
